## Featured Demonstration: `kbrun.mp4`

The primary file in this folder, [`kbrun.mp4`](file:///home/sicmundus/Desktop/Dev/kbworkspaces/project/kernel-borderlands/media/demoruns/kbrun.mp4), is an end-to-end pipeline recording demonstrating how the system detects and traces telemetry events using the live hook verification suite.

[kbtui](https://github.com/user-attachments/assets/6b899472-8873-4265-85af-b41e258fbc40)

> [!NOTE]
> **Hardened Offline-First Architecture**: Kernel Borderlands does not expose or bind any network ports for its internal operations. All communications between subsystems (such as the sensor loader to the daemon, and the daemon to the operator console) are strictly localized to Unix Domain Sockets (UDS) under `/run/kb/`.

> [!IMPORTANT]
> This demonstration shows the baseline sensor, control plane, and TUI integration phase. It does not include the Autonomous Agent Defense Swarm (AADS) or the React Web Dashboard, focusing entirely on verifying the core kernel-to-operator pipeline.

### Scenario Depicted
The demonstration shows the live execution of the [test_all_hooks.sh](file:///home/sicmundus/Desktop/Dev/kbworkspaces/project/kernel-borderlands/kb-core/tests/test_all_hooks.sh) integration script. The script sequentially triggers all 9 telemetry event types and three multi-stage attack chains to test the behavior engine's alert transitions in the console.

### Step-by-Step Walkthrough in the Video
1. **Subsystem Bootstrap**:
   - The user loads the eBPF kernel sensor (`kbd_sensor` in `kb-core`) under root privileges.
   - The Go-based control plane daemon `kbd` is started, initiating the telemetry UDS socket bridge (`/run/kb/kbd.sock`) and the management gRPC UDS socket (`/run/kb/kba.sock`).
   - The Rust/Ratatui-based **Terminal User Interface (`kb-tui`)** is launched locally via `cargo run` and connects to `kbd` over the `/run/kb/kba.sock` UDS interface to monitor live system logs, telemetry counts, and process states.
2. **Executing the Test Suite**:
   - The user runs [test_all_hooks.sh](file:///home/sicmundus/Desktop/Dev/kbworkspaces/project/kernel-borderlands/kb-core/tests/test_all_hooks.sh) in a separate shell pane.
3. **Telemetry Ingestion & State Transitions**:
   - As the script executes, the eBPF sensor intercepts kernel hooks (including local socket binds and connection attempts generated by the script) and forwards them through the `/run/kb/kbd.sock` UDS IPC channel to `kbd`.
   - The behavior engine transitions alert levels for sequential attack chains:
     - **Chain A (Privilege & Credential)**: Transitions the alert state from `SAFE` → `OBSERVED` → `SUSPICIOUS`.
     - **Chain B (Memory Injection)**: Transitions from `SAFE` → `SUSPICIOUS` → `BORDERLANDS` → `COMPROMISED` by allocating RWX memory at address 0.
     - **Chain C (C2 Connection)**: Transitions from `SAFE` → `SUSPICIOUS` → `BORDERLANDS` via outbound socket connection attempts detected at the kernel syscall boundary.
4. **TUI Visualization**:
   - The `kb-tui` console displays the live feed of events, alert statuses, and active process trees changing colors to reflect security states (yellow for SUSPICIOUS, orange for BORDERLANDS, red for COMPROMISED).

## How to Reproduce this Run

### 1. Build and Start Subsystems
Open terminal windows/panes in your development VM and start the following components in order:

**Pane 1: Kernel observation layer (`kb-core`)**
```bash
cd kb-core
make clean && make
sudo ./build/kbd_sensor
```

**Pane 2: Control Plane Daemon (`kb-control-plane`)**
```bash
cd kb-control-plane
go run cmd/kbd/main.go --policy config/policy.yaml
```

**Pane 3: Terminal User Interface (`kb-tui`)**
```bash
cd kb-op/kb-tui
cargo run
```

### 2. Trigger the Threat Simulation
In a separate shell, execute the integration test script:
```bash
cd kb-core
./tests/test_all_hooks.sh
```

Monitor the TUI interface in Pane 3 to verify that the `lsm/file_open`, memory protection, and network socket hooks trigger visual alerts and process zone color changes. All control communication flows entirely over local Unix Domain Sockets.

## Guidelines for Recording New Runs

If you are modifying the behavior engine or sensor hooks, you can record an updated run:

1. **Use a clean terminal setup**: Hide any personalized bash prompts or private network IPs.
2. **Terminal Recorder**: Use `asciinema` to record in text format or OBS Studio configured at 1080p, 30fps.
3. **Compressing Video**: Compress the output MP4 to save repository space using `ffmpeg` before pushing:
   ```bash
   ffmpeg -i raw_run.mp4 -vcodec libx264 -crf 28 -preset slow -acodec aac -b:a 128k kbrun.mp4
   ```
4. **Document the Run**: Update this file with the date of compilation and any changes to the execution order.
